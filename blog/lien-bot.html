LINEで予約が完結する、
ネイルサロンのBotを作った。
— Lien 開発ログ #1

スマホのカメラロールを開かなくていい。電話しなくていい。DM を待たなくていい。
LINEを開いて、写真を見て、ボタンを押すだけで予約が入る。

そういう体験を、架空のネイルサロン「Lien Nail Studio」向けに実装した。

← Blog に戻る

ネイルサロンの予約は、今もDMや電話が多い。 LINEで連絡はできても、空き確認→日程調整→確定という往復が発生する。 お客さんは待たされ、サロン側は都度返信しなければならない。

これは自動化できる。LINEのBot機能を使えば、カルーセルでメニューを見せて、カレンダーで空きを選んで、名前を入れたら予約完了まで全部LINEの中で完結する。

作ったもの

「Lien Nail Studio」という架空のネイルサロン向けLINE予約Botをゼロから実装した。デモとして実際に動いている。

1
写真付きカルーセルでメニュー選択

ワンカラー・ニュアンスデザイン・アート込みコースなど7メニューを写真付きで表示。価格と所要時間も一目でわかる。

2
スタッフ指名

前回来店時の担当を自動で提案。初回なら全スタッフの紹介カードを表示して選べる。

3
1ヶ月先まで空き日程を自動計算

定休日・スタッフ休憩時間・既存予約を考慮して、空き枠だけをボタンで提示する。二重予約は DB レベルで防止。

4
仮受付 → 確定通知

予約は即確定せず「仮受付」として管理画面に届く。サロン側が確認して確定ボタンを押すと、お客様のLINEに通知が届く。

5
前日リマインダー自動送信

予約前日の夜20時に、Cronジョブが自動でリマインドメッセージを送信する。

技術構成

すべて Cloudflare のエコシステムで完結させた。サーバーレスで、コストはほぼゼロ。

— Architecture
Webhook 処理Cloudflare Workers（Edge で動くサーバーレス）
予約データCloudflare D1（SQLite ベースの DB）
セッション管理Cloudflare KV（会話状態を30分保持）
定期実行Cloudflare Cron Triggers（毎日11:00 UTC）
LINE 通信LINE Messaging API（Webhook / Push / Reply）
管理画面Cloudflare Workers（同じ D1 を参照）

LINE Bot の本体は1つの Workers ファイル。Webhook を受け取り、KV から会話ステートを読み、D1 で予約を管理し、LINE API にメッセージを返す。このサイクルが1リクエスト30秒以内に完結しなければならない。

設計で工夫したこと

一番考えたのは会話フローの状態管理だ。LINEのBotはステートレスで、毎回のメッセージが独立したリクエストとして来る。「今このユーザーは予約フローのどのステップにいるか」を保持する仕組みが必要だった。

解決策はシンプルで、Cloudflare KV に `state:{userId}` というキーでJSONを保存する。予約フローに入ったらステートを書き、ステップが進むたびに更新し、完了したら削除する。TTLは30分。タイムアウトしたら自動でリセットされる。

— KV でセッション管理 
// ステート保存（TTL: 30分）
await kv.put(`state:${userId}`, JSON.stringify(state), { expirationTtl: 1800 });

// ステート取得
const raw = await kv.get(`state:${userId}`);
const state = raw ? JSON.parse(raw) : null;

もう一つの工夫が二重予約防止だ。「空き確認 → 選択 → 確定」の間に別ユーザーが同じ枠を予約する競合が起きる可能性がある。D1 の UNIQUE INDEX を使って、同じ日時・スタッフの組み合わせには1件しか入らないようにした。

— UNIQUE INDEX で二重予約を防ぐ 
CREATE UNIQUE INDEX IF NOT EXISTS uniq_slot
  ON reservations(date, start_time, staff_id)
  WHERE status IN ('confirmed', 'pending');

難しかったこと

Cloudflare Workers の30秒制限は最初に躓いた壁だった。LINE の Webhook はリクエストを受けてから30秒以内にレスポンスを返さなければならない。AI画像生成や外部API呼び出しを直列で実行すると簡単に超える。

対策は「まず即レスポンスを返し、重い処理はPromiseで並列実行する」設計に変えることだった。LINEの仕様上、replyTokenで返せるのは1回だけなので、後続メッセージはPush APIで送る。

もう一つはQAキーワードの誤検知問題だ。「ニュアンスデザイン」を選んだとき、「デザイン」というキーワードがQAルールにマッチしてギャラリーの案内を返してしまう。予約フロー中はQAを完全に無効にし、フロー外でのみQAを発火させることで解決した。

— 対応メッセージ種別 5+ テキスト / Flex Message / カルーセル / ボタン / 位置情報
— QAルール数 20+ 料金・営業時間・アクセス・キャンセル規定など

できたもの

動くデモとして公開している。実際にLINEで友だち追加して予約操作が体験できる。スタッフは架空だが、Botの動きは本物だ。

LINEを開く。リッチメニューの「予約」を押す。
写真付きのメニュー一覧が流れてくる。
ワンカラーを選ぶ。スタッフを選ぶ。日付を選ぶ。名前を入れる。
確認画面が出て、「確定」を押す。

それだけで予約が入る。
お客さんは待たなくていい。サロン側は手が空いたときに確認すればいい。
