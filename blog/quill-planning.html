会議を書かないAIを作る
— Quill 開発ログ #1：計画編

ひとりで仕事をしていると、議事録は要らない。
でも社員5人くらいの組織からは、急に必要になる。
そして誰がやっても続かない、そんな話を最近よく聞いた。

だから、それを解く AI を作ってみることにした。

まずLPだけ先に作った

機能を一行も書く前に、LPだけ先に公開した。 名前は「Quill」、羽ペンの意。タグラインは「会議は書かない。」

なぜ先に LP かというと、誰に何を売る AI なのかを、自分の中で固定したかったから。 実装が始まると技術の都合に引きずられて、ユーザー体験がブレる。 だから先にゴールを置いて、それを実装で再現する形にしたい。

LP は quill-static.pages.dev で公開している。 ここから先は、その絵を実物に変えていく作業の話。

計画段階で見えている、6つの問題

「会議を書かない AI」と一行で言うと簡単だが、まじめに作ろうとすると、最初から最後まで問題だらけになる。 今の段階で見えている課題と、それぞれの当たりをつけた解き方を残しておく。 実装が始まれば、たぶんこの計画は半分くらい裏切られる。それも記録の対象。

— Problem 01

そもそも、会議の音声はどう取る？

「マイクで録音すればいい」で終わる話ではない。 その会議に参加してる誰か一人が「録音係」をやる前提だと、絶対に普及しない。 本人が忘れる、設定をミスる、保存先が散らばる。

理想は、会議に AI 自身が参加してくれること。 Zoom や Google Meet で「Quill が参加しました」と通知が出て、終わったら勝手に退出している、という体験。

→ Solve

Bot として会議に参加させる方式を採る。実装オプションは2つ： Recall.ai のような Bot 基盤を借りるか、 Headless ブラウザ × WebRTC で自前実装するか。 MVP では Recall.ai に乗る。自前は絶対にハマる領域。

— Problem 02

文字起こし、Whisper API か、自前ホスティングか？

OpenAI の Whisper API は楽だが、1時間の会議で約 $0.36。 1000ユーザーが平均1日30分使ったら、月 $5,400 がトランスクリプトだけで消える。 ここは早めに設計しておかないと、価格戦略が崩れる。

一方、自前で Whisper をホストすれば固定費にできるが、GPU の管理コストと遅延の制御が新しい問題になる。

→ Solve

MVP は Whisper API 一択でいい。 ユーザーが少ないうちに自前運用は無駄。 月のトークンコストが Pro プランの粗利を食い始めたら、自前ホストへの切り替えを検討。 境目は「同時録音 30本以上が常態化」したあたり。

— Problem 03

「文字起こし」と「議事録」は別物

逐語ログがそのまま届いても、誰も読まない。 「で、結局なに決まったんだっけ？」「私、何やるんだっけ？」に即答できる形でないと、議事録としての価値がない。

つまり 文字起こし → 構造化 → 要約 の3段変換が必要。 ここの精度がプロダクトのコアになる。

→ Solve

多段階プロンプトで処理する：
① 全文から「決定事項」「アクションアイテム」「保留事項」を抽出
② 各アクションに対して「誰が／いつまでに」を逐語ログから埋める
③ 全体を 200〜400字のサマリーに圧縮
各段階で Few-shot で良い議事録の例を見せるのが効く。学習しているフォーマットに引き寄せる。

— Problem 04

話者分離、誰が誰の声か AI に分かる？

diarization（話者分離）は今のモデルなら7-8割は当たる。 ただしこれは「話者A」「話者B」という匿名ラベル。 議事録としては「Aさん」じゃなく「中村さん」と書きたい。

実名を当てるには、別の情報源と突き合わせる必要がある。

→ Solve

Zoom / Meet の 参加者リスト API から名前を取得し、 各話者の発話量と参加者の発言タイミングを照合してマッピングする。 初回はユーザーに「この声は中村さんで合ってる？」と確認させ、 2回目以降は声紋を再利用する。初回だけ少し手間、2回目から自動が現実的。

— Problem 05

リアルタイム vs 会議後、どっちで届ける？

最初は「リアルタイムで議事録が画面に出る」と思っていた。 でもユーザーリサーチを想像してみると、 会議中に議事録画面なんて誰も見ない。話に集中している。

逆に、終わった瞬間に通知が来て、3分後に開いたら全部書かれている、の方が体験として強い。

→ Solve

リアルタイム配信は捨てる。 会議終了の瞬間に処理を開始し、3分以内に Slack / メールへ届ける構成にする。 これならコスト計算も、レイテンシ管理も、UX も全部楽になる。

— Problem 06

「会議を録音されるのは嫌」という心理的ハードル

技術が動いても、文化が動かないと使われない。 「Bot が黙って入ってきて録音している」状態は、たとえ社内の会議でも警戒される。 特にクライアントとの会議では致命的。

→ Solve

Bot 入室時に必ず、「Quill が記録します。停止したい方はホストに伝えてください」と全員にチャットで通知する。 ホスト側にはワンクリック退出ボタンを常時表示。 「録音されている事実が透明である」ことが、心理的ハードルを下げる第一歩。 機能じゃなく、態度の設計。

次回：実装フェーズに入る

ここまでで、機能を作る前に潰しておきたい問題を6つ列挙した。 次回（#2）からは、いよいよコードを書き始める。 最初に手をつけるのは Problem 01 — Bot として会議に参加させる部分。 Recall.ai のドキュメントを読み込みつつ、最小構成のプロトタイプを動かすところから。

予定通りには進まない、というのはわかっている。
Quill を作る過程で、思ってなかった問題が必ず出てくる。
だからこの開発ログは、計画通りに進んだ話ではなく、計画が崩れた瞬間の記録を残す場所にしたい。
